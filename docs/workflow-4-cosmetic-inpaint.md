# Workflow 4 — Cosmetic Inpaint (two-pass bootstrap)

Parent: [comfyui-animation-generation](./pipeline-overview.md). Part of **Pipeline 2 — cosmetic layer generation**. Produces the intermediate video that Workflow 5 (cosmetic segmentation, not implemented yet) will consume.

Takes a base-animation mp4 (output of [Workflow 2](./workflow-2-video-gen.md)) and a still image of the **same character with a cosmetic added**. Produces an mp4 where the cosmetic has been added onto the base character while preserving base motion bit-exact everywhere the cosmetic does not cover.

Uses **Wan 2.1 VACE 14B** (Q5_K_M GGUF) + **CausVid V2 distill LoRA** (`Wan21_CausVid_14B_T2V_lora_rank32_v2.safetensors`) for 8-step inference at `LORA_STRENGTH = 1.0`. V2 is the current default; V1 had saturated colour-highlight drift at high strength that broke tintable greyscale cosmetics.

## The problem this workflow solves

We want pixel-perfect cosmetic layering: base motion preserved exactly outside the cosmetic region, cosmetic painted precisely inside it. This needs a **per-frame mask that covers where the cosmetic is on each frame, and nothing else**. Any mask halo over non-cosmetic pixels gets regenerated by VACE and produces subtle "blending" (hair color bleeding onto skin, colors drifting on body pixels). Any under-coverage produces "clipping" (cosmetic cut off at mask boundary when the character moves past it).

The hard constraint: **the cosmetic reference is a single static image.** A single static 2D mask works for nothing except a perfectly still character — the moment the head tilts or body sways, the mask misaligns and we get clipping or (if dilated) blending.

### Options we explored

1. **Static mask from pixel-diff between cosmetic ref and base frame 1.** Rejected: pixel-diff fails on specular highlights and on cosmetics that don't change silhouette. Also static → fails on motion.
2. **Static mask: SAM3 on frame 1 of base (prompt = `"hair"`) ∪ SAM3 on cosmetic ref (prompt = `"hair"`).** Intended to union "old region" and "new region." Failed because the base character is hairless — SAM3 on frame 1 returned empty — so the union collapsed to just the cosmetic-ref mask (still static). Tested 2026-04-19.
3. **Per-frame SAM3 on base video with prompt = `"hair"`.** Intended to track motion. Failed because the base has no hair to segment — all 33 mask frames came out empty (black). Tested 2026-04-19, VRAM staging worked but output was equivalent to source.
4. **Static cosmetic-ref mask + generous dilation.** The dilation reintroduces the blending failure mode: wherever the dilated halo covers non-cosmetic body pixels, VACE regenerates them from the reference + prompt and produces visible hair-color bleed. Bigger dilation = bigger conflict zone = more visible blending.
5. **Per-frame head-tracking + static cosmetic-ref mask translation.** Feasible but requires landmark detection + custom affine-warp preprocessing. Deferred unless the chosen path fails.
6. **Two-pass bootstrap.** ← **Chosen.** See below.

### Chosen: two-pass bootstrap (pass 2 optional)

The insight that unlocks this: **if the base video had hair, per-frame SAM3 would produce pixel-perfect per-frame masks for free.** So we create a version of the base video with hair — rough is fine — and segment that.

**Pass 1 (rough inpaint).** Run SAM3 on the cosmetic ref only → single static hair mask. Apply dilation. VACE inpaints using this static mask broadcast across all 33 frames. Output is a rough cosmetic video — hair painted approximately where it should be.

**Pass 2 (optional — clean inpaint with precise per-frame masks).** Run SAM3 per-frame on the pass 1 rough video. Because hair now exists in every frame, SAM3 returns a tight, pixel-aligned hair mask per frame. Apply minimal dilation (3px, anti-aliasing only). VACE inpaints the **original base video** using these precise per-frame masks. Non-masked pixels come from the original base via `ImageCompositeMasked` → bit-exact preservation. Masked pixels come from fresh VACE inference → clean hair painted exactly on the hair region.

**Pass 2 is NOT mandatory.** Its role is to tighten the mask around the actual cosmetic pixels, eliminating any blending in the pass-1 dilation halo. If the pass 1 output already looks clean (no visible color bleed from the cosmetic into adjacent skin/body/background), pass 2 adds no value and costs ~1.5–3 min of extra wall time. Use pass 2 **only when you see the cosmetic merging with the background** — e.g. pink hair color tinting nearby skin, halo glow around the silhouette, or any sign that VACE painted outside the true cosmetic region during pass 1.

Rule of thumb:
- Small dilation in pass 1 (≤15 px) and tight-fitting cosmetics → pass 1 is usually enough; skip pass 2 (`RUN_PASS2 = False` in the driver).
- Larger dilation (20–30 px) or loose-fitting cosmetics with visible pass-1 bleed → run pass 2 to re-tighten.

The dilation halo from pass 1, when pass 2 runs, is effectively discarded: pass 2's SAM3 segments only the hair pixels, not the halo-affected pixels around them. The bootstrap works either way — pass 2 is a polish step, not a correctness requirement.

### Options not taken (noted for future iterations)

- **True SAM3 video tracking** (`build_sam3_video_model`). The `ComfyUI-RMBG` node pack only exposes the image model. Would give temporally-consistent masks in one SAM3 call instead of per-frame independent segmentation, reducing edge jitter. Requires a custom node.
- **Per-frame head-tracked affine warp of the static cosmetic-ref mask.** Eliminates the pass 1 inference cost. Requires landmark detection + transform math.
- **Silhouette-dramatic cosmetics** (short hair → waist-length). Neither the two-pass nor any single-reference approach solves this; requires a cosmetic *video* (pose-matched) as input.

## Goal

Given:
- `base.mp4` — a base-animation clip (33 frames, 512×512, 16 fps).
- `cosmetic_ref.png` — a **512×512 PNG** of the character *with* the cosmetic added. Used by SAM3 (pass 1) to derive the static hair mask, and by VACE (both passes) as `reference_image`. Does not need to be pose-aligned with the base video. See [Reference image preparation](#reference-image-preparation-512512-png-required) below — feeding anything else (JPEG, non-512 PNG) silently loses fine-feature signal before VACE runs.

Produce `cosmetic.mp4` — 33 frames, 512×512, 16 fps, identical to `base.mp4` outside the per-frame hair region (pass 2 mask), and with the cosmetic applied inside.

### Reference image preparation (512×512 PNG required)

**Always convert the cosmetic reference to a 512×512 PNG before running the pipeline.** Feeding JPEG or non-512 input works (the workflow's `ImageScale` node downsamples internally), but silently loses fine features — wispy single-pixel strands, thin outlines, low-contrast details — before VACE or SAM3 sees them. VACE cannot paint what's already been smoothed away.

**Measured example (2026-04-20):** the original `Add_pink_anime_202604192212.jpeg` (1024×1024 JPEG) had clearly visible wispy face-framing strands crossing the cheeks. At 512×512 (lanczos downsample of the JPEG) those strands were largely gone. No LoRA/CFG/sampler/seed combination we tried recovered them, because the signal was erased upstream of the model. Running the pipeline with a `512×512 PNG` pre-converted from the same JPEG produced visually identical output — confirming the loss happens during the 2× lanczos downsample, not inside the model.

**Preparation steps:**

1. **Render the reference at 512×512 natively** when possible. If you're generating the reference via Qwen-Image-Edit (Workflow helper) or any other image model, set the output resolution to 512×512 instead of 1024 and downsampling afterward. This is the only way to guarantee no downsample detail loss.
2. **Save as PNG** (lossless). JPEG's DCT quantization + chroma subsampling erases low-contrast fine detail regardless of resolution.
3. **If you must start from a higher-resolution or JPEG source**, downsample once with `lanczos` then apply a light unsharp mask to restore thin-feature visibility, and save as PNG. Open the 512 PNG in an image viewer and visually confirm the target cosmetic detail (strands, outlines, etc.) is preserved before running the pipeline. If a feature isn't visible at 512×512, the pipeline cannot produce it.
4. **Feed the 512×512 PNG directly as `COSMETIC_REF`** in the driver. The `ImageScale` node in the workflow becomes a no-op for 512 input (kept in the graph for flexibility with non-512 input, but should never actually downsample for a production run).

## Implementation

**Workflow files:**
- `F:/ComfyUI/user/default/workflows/w4-cosmetic-inpaint/pass1_rough_api.json` — pass 1.
- `F:/ComfyUI/user/default/workflows/w4-cosmetic-inpaint/pass2_clean_api.json` — pass 2.

**Driver:** `F:/ComfyUI/user/run_w4_cosmetic_inpaint.py` — orchestrates both passes. Run with `F:/ComfyUI/.venv/Scripts/python.exe F:/ComfyUI/user/run_w4_cosmetic_inpaint.py`.

### Eviction discipline (unchanged across both passes)

SAM3 (~4 GB) and VACE DiT (~13 GB) must never coexist in VRAM. Within each pass, three mechanisms combine to guarantee this:

1. **`SAM3Segment` with `unload_model=true`** evicts SAM3 internally and calls `torch.cuda.empty_cache()` before returning.
2. **KSampler input-dict ordering** — `positive`, `negative`, `latent_image` are listed **before** `model`. ComfyUI's executor walks inputs in dict insertion order, so the conditioning chain (which transitively triggers SAM3) runs first; the model chain (UnetLoaderGGUF → LoRA → ModelSamplingSD3) runs last. This single mechanism is the guarantee SAM3 has already evicted before VACE DiT loads.
3. **`VRAMUnloadClip`** evicts UMT5 after text encode. **`VRAMUnloadModel`** evicts VACE DiT before VAE decode.

**Between passes:** the driver POSTs `/free` (`{"unload_models": true, "free_memory": true}`) to ComfyUI to purge any residual models. Combined with the in-pass discipline above, this gives the same strict "SAM3 xor VACE" invariant for the pass-2 SAM3 call.

### Pass 1 — rough inpaint (node graph overview)

Purpose: produce a video with hair painted approximately in the right places, to feed pass 2's SAM3.

1. `LoadVideo` (base) → `GetVideoComponents` → `ImageScale` → scaled source batch `[33, 512, 512, 3]`.
2. `LoadImage` (cosmetic ref) → `ImageScale` → `[1, 512, 512, 3]`.
3. `SAM3Segment` on **cosmetic ref only** (single image), `prompt = COSMETIC_DESCRIPTION`, `unload_model = true` → static mask `[1, 512, 512]`.
4. `AILab_MaskEnhancer` with `mask_offset = 15` (default; per-use-case knob), `smooth = 2`, `fill_holes = true`.
5. `SaveImage` of the enhanced mask → `masks/pass1_static_mask_NNNNN_.png`.
6. `CLIPLoaderGGUF` → `CLIPTextEncode` × 2 → `VRAMUnloadClip`.
7. `VAELoader` → `WanVaceToVideo` (length=33, `control_video` = scaled source, `control_masks` = static mask [1,512,512] — VACE broadcasts to all frames, `reference_image` = scaled cosmetic ref).
8. `UnetLoaderGGUF` → `LoraLoaderModelOnly` (CausVid, 0.3) → `ModelSamplingSD3` (shift=5) → `KSampler` (steps=4, cfg=1, uni_pc/simple).
9. `VRAMUnloadModel` → `TrimVideoLatent` → `VAEDecode` → `ImageCompositeMasked` (destination = scaled source, source = inpainted, mask = static) → `CreateVideo` (fps=16) → `SaveVideo` → `pass1/<cosmetic>_<animation>_pass1_NNNNN_.mp4`.

### Pass 2 — clean inpaint (node graph overview)

Purpose: produce the final deliverable by inpainting the original base video with precise per-frame masks segmented from the pass 1 output.

1. `LoadVideo` (base) → `GetVideoComponents` → `ImageScale` → scaled source batch `[33, 512, 512, 3]` (used as control_video + composite destination).
2. `LoadVideo` (pass 1 rough) → `GetVideoComponents` → `ImageScale` → scaled pass 1 batch `[33, 512, 512, 3]` (used only as SAM3 input).
3. `LoadImage` (cosmetic ref) → `ImageScale` → `[1, 512, 512, 3]` (VACE reference_image only).
4. `SAM3Segment` on **scaled pass 1 batch** (all 33 frames), `prompt = COSMETIC_DESCRIPTION`, `unload_model = true` → per-frame mask batch `[33, 512, 512]`.
5. `AILab_MaskEnhancer` with `mask_offset = 3` (default; per-use-case knob), `smooth = 1`, `fill_holes = true`.
6. `SaveImage` of the enhanced masks → 33× `masks/pass2_precise_mask_NNNNN_.png`.
7. `CLIPLoaderGGUF` → `CLIPTextEncode` × 2 → `VRAMUnloadClip`.
8. `VAELoader` → `WanVaceToVideo` (length=33, `control_video` = scaled **base** source, `control_masks` = precise per-frame batch, `reference_image` = cosmetic ref).
9. `UnetLoaderGGUF` → LoRA → `ModelSamplingSD3` → `KSampler`.
10. `VRAMUnloadModel` → `TrimVideoLatent` → `VAEDecode` → `ImageCompositeMasked` (destination = scaled **base** source, source = inpainted, mask = precise per-frame) → `CreateVideo` → `SaveVideo` → `pass2/<cosmetic>_<animation>_pass2_NNNNN_.mp4`.

### VRAM staging (per pass, projected)

Both passes have the same residency profile.

| Phase | Resident in VRAM | Size |
|---|---|---|
| SAM3 load + segmentation (pass 1: 1 image; pass 2: 33-frame batch) | SAM3 | ~4 GB |
| After SAM3 returns (`unload_model=true`) | — (almost empty) | <0.5 GB |
| UMT5 load + text encode | UMT5 | ~5 GB |
| After `VRAMUnloadClip` | — (almost empty) | ~0.5 GB |
| VAE load + `WanVaceToVideo` build | VAE | ~0.5 GB |
| After VACE DiT + LoRA load | VACE DiT + VAE | ~14 GB |
| Peak during KSampler 4 steps | VACE DiT + activations | ~16 GB |
| After `VRAMUnloadModel` | VAE only | ~0.5 GB |
| VAE decode peak | VAE + decode buffer | ~3 GB |

Projected peak per pass: **~16 GB** on a 24 GB card. Between passes, `/free` brings resident memory back to baseline (~2–3 GB).

### CausVid LoRA strength — detail vs. silhouette tradeoff (measured 2026-04-20 on V1)

CausVid distillation LoRA strength controls how hard VACE is steered toward clean, structurally-correct single-pass outputs. We swept it on the pink_hair + idle_breathing_5 test case **using CausVid V1**:

| `LORA_STRENGTH` | Result |
|---|---|
| **0.1** | Structural collapse — VACE paints only a short pink cap on the head, no silhouette. Distillation pull too weak to commit at 6 steps. |
| **0.3** (V1 template default) | Clean shoulder-length curly silhouette, bangs visible. Missing fine face-framing strands crossing the cheeks. |
| **0.4** | Visually indistinguishable from 0.3. |
| **0.6** | Slightly more visible internal strand separation within the silhouette. Missing face-framing strands still absent. |
| **0.8** | More vibrant saturation, more prominent internal strand highlights, subtle halo artifact around character edges. Face-framing strands still absent. |

**Reading:** LoRA strength controls **internal texture detail** (strand separation, highlights) inside whatever silhouette the LoRA commits to. It does **not** control silhouette-level features — if a detail lives outside the silhouette the LoRA has committed to (e.g. wispy strands crossing the cheeks), no strength setting will produce it. Different seeds may roll different silhouette commitments.

**This sweep is V1-specific. With CausVid V2 (current default) the silhouette/strength relationship is similar, but the colour behaviour is different — V1 at high strength (0.8–1.0) injected saturated colour highlights (green/blue/yellow/black) regardless of prompt, breaking tintable greyscale cosmetics; V2 at 1.0 produces uniformly white/grey output. Recommended V2 default: `LORA_STRENGTH = 1.0`, `PASS1_STEPS = 8`. Drop to 0.5–0.7 if you need to ship a cosmetic in colour and the V2 colour fidelity isn't tight enough — but for tintable greyscale, 1.0 is the right choice.**

### Pass 1 steps — fine detail

`PASS1_STEPS = 8` is the V2 recommended default (V1 used 6, which was already up from CausVid's standard 4). Pass 1 has to paint the cosmetic from scratch; extra steps give the sampler more room to commit to fine features. `PASS2_STEPS` stays at 4 because pass 2 is structural replacement inside a precise mask, not fine-detail generation.

### Pass 1 positive prompt — keep it short

**Less is more.** The positive prompt in pass 1 visibly influences texture density inside the silhouette but **not** silhouette shape. Long descriptive prompts (multi-line "structural strands, sharp tips, layered bangs, ...") didn't help and sometimes hurt — over-specifying colour ("pure white", "snow white") was correlated with V1's saturated colour drift. A short prompt focused on character + style + cosmetic ("cute chibi character with short messy spiky white anime style hair, crisp clean lineart") plus a minimal negative prompt (artifacts only, no per-colour negatives) produces the cleanest output with V2.

**Anti-pattern** (don't do): listing every off-colour in the negative prompt to "block" V1's colour drift. It doesn't work — V1 ignores the negatives and paints saturated highlights anyway. Switch to V2 instead.

### Mask dilation — per-use-case tuning

Dilation values are exposed as driver knobs because the right value depends on:

- **Character motion envelope:** near-static animations (idle_breathing) tolerate smaller dilation; walking/jumping needs more.
- **Cosmetic silhouette vs. base silhouette:** a cosmetic that extends beyond the base region (long hair on a short-haired base) needs more pass-1 dilation to cover the extension; a cosmetic that fits within the base silhouette can use less.
- **Character resolution / line thickness:** chibi characters with thick outlines benefit from slightly more pass-2 dilation to avoid cutting outline pixels.

**Defaults:**

| Knob | Default | When to raise | When to lower |
|---|---|---|---|
| `PASS1_MASK_DILATE` | **15 px** | Cosmetic visibly cut off in pass 1 output (SAM3 will then miss those hair pixels too). | Pass 1 bleeds hair far from the character (SAM3 may mis-segment halo artifacts). |
| `PASS2_MASK_DILATE` | **3 px** | Cosmetic edge clipped in pass 2 output (visible hard seam on composite). | Visible halo blending in pass 2 output (non-hair pixels tinted). |

Pass 1 dilation does **not** directly appear in the final deliverable — it only affects what region VACE gets to paint during the bootstrap. Pass 2 dilation appears in the final output's composite boundary. So pass 2 is the sensitive knob; pass 1 has more tolerance.

### Detail fidelity — improvement options to try (researched 2026-04-20, none tested yet)

Background: seed sweeps (9 seeds, measured 2026-04-20 on pink_hair + walking_right) all produced the same simplified silhouette — forehead bangs + side curls, no wispy strands crossing the cheeks that are present in the reference. This matches the documented VACE behaviour that `reference_image` is **loose guidance, not strict visual preservation**, and is compounded by CausVid V1's distillation which further simplifies features. Below are the community-recommended paths to improve fidelity without dropping CausVid. Ordered by expected leverage.

1. **Upgrade to CausVid V2** (`Wan21_CausVid_14B_T2V_lora_rank32_v2.safetensors` from Kijai/WanVideo_comfy, ~205 MB on disk — *smaller* than V1's 319 MB, so no extra VRAM cost; peak stays ~17.6 GB). Community reports V2 quality is *"almost identical to base Wan 2.1"* at 8 steps. Bump `PASS1_STEPS` from 6 → 8 when swapping. Single highest-leverage change, and the cheapest (file swap + one knob). **Tested 2026-04-20 on pink_hair + walking_right, seed 12345, LoRA strength 0.8, 8 steps:** wall time 213 s (vs. V1's 181 s at 6 steps). Silhouette commitment unchanged — forehead bangs + side curls, still no face-framing strands crossing the cheeks. Minor visible improvements: slightly more feathered/detailed bangs, subtly more vibrant color, cleaner lineart. **V2 alone does not break the silhouette commitment for this cosmetic.** Stack with option 2 (dual-sampler) and/or option 3 (strength 1.0) for another shot.
2. **Dual-sampler CFG trick.** Kijai's V2 README explicitly notes V2 *"needs more steps and can also benefit from cfg"* — meaning CFG > 1 is supported (unlike V1 which is locked to CFG 1 by CausVid distillation). Community workflow from discussion #27 (funscripter): total 10 steps split into two `KSamplerAdvanced` nodes — sampler #1 runs steps 0→4 at CFG 4 (pulls the model toward reference fidelity in the high-signal early steps), sampler #2 runs steps 4→10 at CFG 1 (CausVid-compatible). Recovers silhouette detail the pure-CFG-1 sampler skips. Requires a workflow-JSON edit (two KSampler nodes + shared latent + pass-through `return_with_leftover_noise`). Pairs well with option 1. **Tested 2026-04-20 on pink_hair + walking_right, V2, strength 1.0, 10 steps total (split 4+6), CFG 4→1, seed 12345:** wall time 237 s (+13 s vs. single-sampler at same total steps). Silhouette commitment still unchanged — no face-framing strands. Visually very similar to the single-sampler strength 1.0 output. **Dual-sampler at CFG 4 does not break the silhouette commitment for this cosmetic.** Higher sampler-1 CFG (6–8) could be tried but likely has diminishing returns given the progression 1.0 → 4.0 produced no silhouette change.
3. **Try a higher `LORA_STRENGTH` (0.9–1.0).** No canonical recommendation exists — Kijai's README gives no explicit number, and community values across discussions #27/#30 span 0.3 (funscripter dual-sampler), 0.53 (chenxiYan), 0.7–1.0 *"your eyes will tell you"* (MavenDE), and 0.95 (rahul7star). Our sweep stopped at 0.8 (chibi-scale halo tolerance); the V1 flashing-at-full-strength issue is fixed in V2 per Kijai's README, so 0.9–1.0 is safe to try now. Sweep 0.9 and 1.0 against our 0.8 baseline — accept slightly more halo in the composite if it gains silhouette detail. **Tested 2026-04-20 on pink_hair + walking_right, V2, strength 1.0, 10 steps, seed 12345:** wall time 224 s. Silhouette commitment still unchanged — no face-framing strands. Minor visible improvement: slightly more vibrant/saturated color, slightly more internal strand definition. **Strength 1.0 + 10 steps alone does not break the silhouette commitment.**
4. **`PASS1_STEPS = 10–20`.** Community consensus *"4–20 steps, 10 gives very good results."* Pass 1 is the silhouette-commitment pass; extra steps give the sampler more room to commit to fine features. Cost: ~+40 s wall-time per extra 4 steps on a 3090. Pair with option 3 (higher strength) rather than replace it.
5. **Stack a Detail Enhancer LoRA** on top of CausVid (as done in community "VACE Cocktail" workflows). Requires sourcing a compatible enhancer LoRA (e.g. identity-preservation or texture-detail LoRAs from Civitai). Adds VRAM (another ~200–500 MB) and a second LoRA load node. Untested for our chibi use case.
6. **Re-generate the cosmetic reference with structural (not wispy) strands.** VACE commits to structural features far more reliably than wispy ones. Regenerate the pink_hair reference with thicker, anchored-to-hairline strands via Qwen-Image-Edit pose workflow. Changes the cosmetic, not the pipeline — cheapest behavioral change but shifts the target design.

**Recommended order if trying:** option 1 alone first (~3 min/pass 1, one file swap). If insufficient, add option 2 (dual-sampler) + option 3 (strength 1.0). Skip 4 unless still insufficient; skip 5 unless we find a validated enhancer LoRA; skip 6 unless design review approves the stylistic change.

**Fundamental floor:** even the vanilla Wan 2.1 VACE teacher model treats `reference_image` as loose guidance — some silhouette-level simplification is inherent to the architecture, not just CausVid distillation. Options 1–5 raise the fidelity ceiling but cannot eliminate it entirely.

Note: if the strands aren't in the pass 1 video to begin with, lowering confidence doesn't help — SAM3 can't segment what isn't there. Verify by sampling the pass 1 output first. |

## Wall-time (measured 2026-04-20 on 3090)

| Pass | Wall time | Peak VRAM |
|---|---|---|
| Pass 1 (cold start) | **167 s** | **~17.6 GB** |
| Pass 2 (after `/free`) | **169 s** | **~17.6 GB** |
| **End-to-end** | **~5.6 min** | well under 24 GB |

Pass 2 was **not faster** than pass 1 despite running immediately after. This is the cost of the `/free` between passes: it purges ComfyUI's model tracker and CPU-RAM GGUF caches, so pass 2 reloads UMT5 + VACE DiT from disk the same as pass 1. Trading ~80 s of wall-time for the strict "SAM3 never coexists with VACE" invariant is the right call given this workflow's memory pressure, but if a future 32+ GB card relaxes that invariant, dropping `/free` between passes would cut pass 2 to ~60–90 s.

### VRAM behaviour observed

Sampled every 10–15 s during the test:
- Baseline after startup `/free`: 2.7 GB
- SAM3 active (pass 1 on single image): 5.8 GB peak → 4.7 GB after `unload_model=true`
- UMT5 load + encode: ~8–11 GB (overlaps briefly with SAM3 residue in PyTorch caching allocator, but true SAM3 weights are gone)
- VACE DiT load + KSampler: **17.6 GB peak** during sampling
- Post-`VRAMUnloadModel`: ~3 GB
- Between-pass `/free`: back to 2.7 GB baseline
- Pass 2 repeated the same profile

No OOM, no coexistence of SAM3 and VACE DiT.

## Troubleshooting

### Pass 1 mask is empty / all black
SAM3 couldn't find the described region on the cosmetic reference. Check that `COSMETIC_DESCRIPTION` matches what's actually visible (e.g. `"hair"` — not `"pink hair"` — because SAM3 segments concepts, not colors). Lower `SAM_CONFIDENCE` to 0.2 for stylized art.

### Pass 1 output shows hair far from the character
Pass 1 dilation too generous. Reduce `PASS1_MASK_DILATE` (try 8–10). If the cosmetic is genuinely large (extends far beyond base silhouette), this is expected — pass 2 SAM3 will still segment only the actual hair pixels.

### Pass 2 mask is empty / all black
Pass 1 didn't paint recognizable hair. Check the pass 1 output mp4 visually — if hair is missing or extremely faint, increase `PASS1_MASK_DILATE`, check reference image quality, or verify the cosmetic is visible and well-defined in the reference. Also verify `COSMETIC_DESCRIPTION` matches the reference content.

### Pass 2 output has visible hard seam on composite
`PASS2_MASK_DILATE` too small. Raise to 5–7.

### Pass 2 output has visible halo / hair bleeding onto body
`PASS2_MASK_DILATE` too large. Reduce to 1–2. If it's already 1–2 and still blending, increase `MASK_SMOOTH` instead of `MASK_DILATE` — softer edges reduce the composite seam without expanding the inpaint region.

### Pass 1 output shows cosmetic merging with background / halo glow
This is the specific signal that pass 2 is worth running. Enable it via `RUN_PASS2 = True` — pass 2 will re-segment the actual cosmetic pixels from the rough video and produce a tight per-frame mask that excludes the blended halo.

### Hair silhouette looks simplified (missing fine strands / face-framing curls)
Not a bug — CausVid LoRA commits to a simplified silhouette at distillation. See [CausVid LoRA strength section](#causvid-lora-strength--detail-vs-silhouette-tradeoff-measured-2026-04-20). Bumping `LORA_STRENGTH` up to 0.8 adds internal strand texture but not new silhouette features. Try a different `SEED` — silhouette commitment varies run-to-run, and a lucky seed may include the missing strands without any config change.

### OOM during pass 1 or pass 2 VACE DiT load
KSampler input-order trick failed in that pass. Driver posts `/free` before each pass, so this shouldn't recur across passes, but within a pass the in-workflow trick is the only line of defense. Fallback: split the problematic pass itself into two prompts (mask generation as one prompt → SaveImage → second prompt loads the PNG batch and runs VACE). Not implemented.

### VACE DiT cold-load takes >60 s
Normal for pass 1. Should be <15 s on pass 2 if ComfyUI's CPU-RAM cache kicks in.
